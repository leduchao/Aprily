# Chat Feature Spec

## Goal

Build direct 1-to-1 chat for authenticated users. The first backend version
will support creating or reusing a direct conversation, sending messages,
listing the current user's conversations, reading a conversation's messages,
and marking a conversation as read.

Group chat, attachments, reactions, edits, deletes, typing indicators, and
real-time delivery are out of scope for this iteration.

## Existing Project Constraints

- The backend follows a Vertical Slice style under `Features/<Feature>/UseCases`.
- Public user identifiers exposed by the API are `users.entity_id` UUID values.
- Database relationships should use internal integer `users.id` keys.
- API responses should use public UUID identifiers, not internal integer IDs.
- Commands/queries should return `Result` or `Result<T>`.
- Read-heavy queries may use Dapper through `IDbConnectionFactory`.
- Write workflows may use `AppDbContext`.
- Endpoints should be mapped from `Features/Chat/ChatEndpoints.cs`.
- Chat endpoints must require bearer authentication.

## Database Design

Add the following tables to `src/Aprily.Backend/Database/InitDatabase.sql`.

### `conversations`

Stores a chat thread. For now, every row is a direct conversation.

Columns:

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `INTEGER GENERATED BY DEFAULT AS IDENTITY` | yes | Primary key |
| `type` | `VARCHAR(20)` | yes | Default `direct`; constrained to `direct` |
| `last_message_id` | `INTEGER` | no | Nullable FK to `messages(id)` added after `messages` exists |
| `last_message_at` | `TIMESTAMPTZ` | no | Used for conversation ordering |
| `entity_id` | `uuid` | yes | Public conversation id, default `gen_random_uuid()` |
| `created_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `updated_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `is_deleted` | `BOOLEAN` | yes | Soft delete flag, default `false` |

Indexes:

- `ux_conversations_entity_id` unique on `entity_id`
- `ix_conversations_last_message_at` on `last_message_at DESC`

### `conversation_members`

Stores user membership in a conversation.

Columns:

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `INTEGER GENERATED BY DEFAULT AS IDENTITY` | yes | Primary key |
| `conversation_id` | `INTEGER` | yes | FK to `conversations(id)` |
| `user_id` | `INTEGER` | yes | FK to `users(id)` |
| `last_read_message_id` | `INTEGER` | no | FK to `messages(id)` added after `messages` exists |
| `last_read_at` | `TIMESTAMPTZ` | no | Last time user marked the conversation read |
| `entity_id` | `uuid` | yes | Public membership id, default `gen_random_uuid()` |
| `created_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `updated_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `is_deleted` | `BOOLEAN` | yes | Soft delete flag, default `false` |

Constraints and indexes:

- `fk_conversation_members_conversations_conversation_id`
- `fk_conversation_members_users_user_id`
- `ux_conversation_members_conversation_id_user_id` unique on active membership pair
- `ix_conversation_members_user_id`
- `ix_conversation_members_conversation_id`

### `direct_conversations`

Ensures there can only be one direct conversation between the same two users.
The smaller internal user id is stored in `user_low_id`, the larger in
`user_high_id`.

Columns:

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `INTEGER GENERATED BY DEFAULT AS IDENTITY` | yes | Primary key |
| `conversation_id` | `INTEGER` | yes | Unique FK to `conversations(id)` |
| `user_low_id` | `INTEGER` | yes | FK to `users(id)` |
| `user_high_id` | `INTEGER` | yes | FK to `users(id)` |
| `entity_id` | `uuid` | yes | Public row id, default `gen_random_uuid()` |
| `created_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `updated_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `is_deleted` | `BOOLEAN` | yes | Soft delete flag, default `false` |

Constraints and indexes:

- `ck_direct_conversations_different_users`: `user_low_id < user_high_id`
- `ux_direct_conversations_user_pair` unique on `(user_low_id, user_high_id)`
- `ux_direct_conversations_conversation_id` unique on `conversation_id`

### `messages`

Stores messages in a conversation.

Columns:

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `INTEGER GENERATED BY DEFAULT AS IDENTITY` | yes | Primary key |
| `conversation_id` | `INTEGER` | yes | FK to `conversations(id)` |
| `sender_user_id` | `INTEGER` | yes | FK to `users(id)` |
| `content` | `VARCHAR(4000)` | yes | Trimmed non-empty text |
| `sent_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `entity_id` | `uuid` | yes | Public message id, default `gen_random_uuid()` |
| `created_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `updated_at` | `TIMESTAMPTZ` | yes | Default `NOW()` |
| `is_deleted` | `BOOLEAN` | yes | Soft delete flag, default `false` |

Constraints and indexes:

- `fk_messages_conversations_conversation_id`
- `fk_messages_users_sender_user_id`
- `ck_messages_content_not_empty`
- `ix_messages_conversation_id_sent_at_id` on `(conversation_id, sent_at DESC, id DESC)`
- `ix_messages_sender_user_id`

After creating `messages`, add nullable FK constraints:

- `conversations.last_message_id` references `messages(id)`
- `conversation_members.last_read_message_id` references `messages(id)`

## API Contract

All chat endpoints are under `/api/v1/chat` and require authentication.

### `GET /api/v1/chat/conversations`

List conversations for the current user.

Query:

| Name | Type | Default | Notes |
| --- | --- | --- | --- |
| `take` | integer | `20` | Range `1..50` |
| `before` | DateTimeOffset? | null | Cursor based on `lastMessageAt` |

Response item:

```json
{
  "id": "conversation-uuid",
  "type": "direct",
  "otherUser": {
    "id": "user-uuid",
    "username": "april",
    "fullName": "April Ly",
    "avatarUrl": null
  },
  "lastMessage": {
    "id": "message-uuid",
    "senderUserId": "user-uuid",
    "content": "Hello",
    "sentAt": "2026-06-18T10:00:00Z"
  },
  "lastMessageAt": "2026-06-18T10:00:00Z",
  "unreadCount": 2
}
```

Ordering: newest `last_message_at` first.

### `GET /api/v1/chat/conversations/{conversationId}/messages`

Read messages in a conversation if the current user is a member.

Query:

| Name | Type | Default | Notes |
| --- | --- | --- | --- |
| `take` | integer | `30` | Range `1..100` |
| `before` | DateTimeOffset? | null | Cursor based on `sentAt` |

Response item:

```json
{
  "id": "message-uuid",
  "conversationId": "conversation-uuid",
  "senderUserId": "user-uuid",
  "senderUsername": "april",
  "senderAvatarUrl": null,
  "content": "Hello",
  "sentAt": "2026-06-18T10:00:00Z",
  "isMine": true
}
```

Ordering: newest first. The client may reverse the list for chronological
display.

### `POST /api/v1/chat/direct-messages`

Send a direct message to another user. If the direct conversation does not
exist, create it.

Request:

```json
{
  "recipientUserId": "recipient-user-uuid",
  "content": "Hello"
}
```

Validation:

- `recipientUserId` must exist and must not be the current user.
- `content` is required after trimming.
- `content` max length is `4000`.

Response:

```json
{
  "conversationId": "conversation-uuid",
  "message": {
    "id": "message-uuid",
    "conversationId": "conversation-uuid",
    "senderUserId": "current-user-uuid",
    "senderUsername": "april",
    "senderAvatarUrl": null,
    "content": "Hello",
    "sentAt": "2026-06-18T10:00:00Z",
    "isMine": true
  }
}
```

Persistence behavior:

- Find or create the direct conversation for the current user and recipient.
- Insert both conversation members when creating a conversation.
- Insert the message.
- Update `conversations.last_message_id` and `conversations.last_message_at`.

Concurrency behavior:

- The unique `(user_low_id, user_high_id)` constraint is the source of truth.
- If two requests create the same direct conversation at the same time, handle
  the unique violation by loading the existing conversation and retrying the
  message insert.

### `POST /api/v1/chat/conversations/{conversationId}/read`

Mark a conversation as read for the current user.

Request:

```json
{
  "messageId": "message-uuid"
}
```

Rules:

- Current user must be a member of the conversation.
- `messageId` must belong to the same conversation.
- Update `conversation_members.last_read_message_id` and `last_read_at`.
- If `messageId` is omitted in a later implementation, default to the current
  conversation `last_message_id`. For this iteration, keep it required.

Response:

```json
{
  "conversationId": "conversation-uuid",
  "lastReadMessageId": "message-uuid",
  "lastReadAt": "2026-06-18T10:00:00Z"
}
```

## Endpoint Mapping

`Program.cs` must call:

```csharp
app.MapChatEndpoints();
```

`ChatEndpoints.cs` should map:

```text
GET  /api/v1/chat/conversations
GET  /api/v1/chat/conversations/{conversationId}/messages
POST /api/v1/chat/direct-messages
POST /api/v1/chat/conversations/{conversationId}/read
```

## Error Codes

Use these error codes for predictable client handling:

| Code | Meaning |
| --- | --- |
| `chat.recipient_not_found` | Recipient user does not exist |
| `chat.cannot_message_self` | User tried to message themself |
| `chat.conversation_not_found` | Conversation does not exist or user is not a member |
| `chat.message_not_found` | Message does not exist or is outside the conversation |
| `chat.message_empty` | Message content is empty |
| `chat.message_too_long` | Message content exceeds max length |
| `chat.send_failed` | Unexpected send/concurrency failure |

## Implementation Order

1. Update `InitDatabase.sql` with chat tables, constraints, and indexes.
2. Add chat entities and update `AppDbContext`.
3. Add response models.
4. Implement `SendDirectMessage`.
5. Implement `ListConversations`.
6. Implement `GetConversationMessages`.
7. Implement `MarkConversationAsRead`.
8. Map chat endpoints in `ChatEndpoints.cs` and `Program.cs`.
9. Add focused tests for direct conversation creation, duplicate conversation
   prevention, message send validation, and membership authorization.

## Acceptance Criteria

- An authenticated user can send a message to another existing user.
- Sending the first direct message creates exactly one direct conversation.
- Sending more messages between the same two users reuses the conversation.
- Users can only list and read conversations where they are members.
- Conversation list includes other user info, last message, and unread count.
- Message history is paginated and excludes soft-deleted messages.
- Mark-as-read updates the current user's membership only.
- All chat API identifiers are UUID public IDs.
